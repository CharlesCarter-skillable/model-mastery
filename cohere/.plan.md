# Cohere on Microsoft Foundry — 3‑Hour Workshop: Reusable Implementation Plan

> This plan is the **authoritative spec** for the workshop that lives at `cohere/` in this
> repository. It is written so that a different agent, on a different device or Azure
> account, can re-implement the workshop end-to-end from scratch and arrive at the same
> shape, conventions, and behaviour. Every section captures both the **what** and the
> **why** so the next implementer can deviate intelligently when needed.

---

## 1. Workshop goals and audience

- **Duration**: 3 hours, self-guided.
- **Audience**: technical (L200–L400). Assume strong programming skills but **new** to
  Microsoft Foundry, Cohere models, embeddings, rerank, AI agents, and AI evaluation.
- **Outcome**: by the end the learner can (1) deploy Cohere models on Microsoft Foundry,
  (2) build, evaluate, red-team and monitor an agentic app with Command A, and (3)
  choose the right Cohere model for the right task (embed / rerank / multimodal).
- **Story**: **enterprise travel concierge** — books cars/flights/hotels and honours a
  corporate travel policy. The story runs through Lab 1, with Lab 0 setting up the
  factory and Lab 2 deep-diving into Cohere's other specialised models.

## 2. Naming and product conventions (must follow)

- The product is **Microsoft Foundry**. Never write "Azure AI Foundry".
- Prefer the **new Foundry SDK** (`azure-ai-projects`) wherever possible. Fall back to
  the Cohere SDK only when needed (e.g., `cohere.ClientV2` for the upstream Cohere
  notebooks that rely on its native call shapes).
- Auth: prefer `DefaultAzureCredential` (AAD). Account API keys are exposed as env
  vars only because the upstream vendor notebooks use them.
- The folder lives at `cohere/`. Sibling provider folders (`anthropic/`, `foundry/`) are
  separate workshops outside this plan's scope.

## 3. Authoritative model deployment facts

The four model deployments learners create (and which the reference Foundry project
`nitya-modelmastery-cohere` in `eastus2` already has) are:

| Deployment name           | Catalog model            | Version | Format | sku.name        | sku.capacity |
| ------------------------- | ------------------------ | ------- | ------ | --------------- | ------------ |
| `text-embedding-3-small`  | `text-embedding-3-small` | `1`     | OpenAI | `GlobalStandard`| 120          |
| `command-a`               | `cohere-command-a`       | `1`     | Cohere | `GlobalStandard`| 1            |
| `Cohere-rerank-v4.0-pro`  | `Cohere-rerank-v4.0-pro` | `1`     | Cohere | `GlobalStandard`| 1            |
| `embed-v-4-0`             | `embed-v-4-0`            | `1`     | Cohere | `GlobalStandard`| 1            |

Foundry resource shapes:
- Foundry account = `Microsoft.CognitiveServices/accounts` with `kind=AIServices`,
  `sku.name=S0`.
- Foundry project = `Microsoft.CognitiveServices/accounts/projects` child resource.
- Account endpoint pattern: `https://<account>.services.ai.azure.com/`.
- Project endpoint pattern (for `azure-ai-projects`):
  `https://<account>.services.ai.azure.com/api/projects/<project>`.
- Cohere ClientV2 `base_url` pattern: `https://<account>.services.ai.azure.com/providers/cohere`
  (ClientV2 auto-appends `/v2/embed` or `/v2/rerank`).

Region default: **eastus2** (validated to host all four models above).

## 6. Notebook style conventions (mandatory, every notebook)

Every notebook (including the vendor 2a–2e) must conform to this outer frame, with
only minimal sanitization of the vendor body:

1. A one-line **time-estimate badge** at the very top, e.g. `> ⏱ ~12 min`.
2. **Section 1** = `# Scenario` markdown that sets the stage.
3. **Section 2** = `## 2. What you will do` numbered task list, **followed by a `---`
   horizontal rule** so it is visually clear where the lab body begins.
4. Body sections numbered sequentially. Alternate markdown and code cells; markdown
   teaches intent and gives analogies, code carries meaningful inline comments.
5. **Final section** = `## N. What you learned` with **exactly 3 takeaways**.

Additional voice rules:
- Write for a high-school reading level. Short sentences, common words, define
  terms on first use in each file (embedding, rerank, evaluator, prompt agent,
  deployment, telemetry, red-team, RPS, App Insights, Foundry project, …).
- Use the established analogies — car-factory for Foundry, travel-concierge for the
  Lab 1 agent. Do **not** add new travel analogies inside the vendor 2a–2e bodies.
- Code comments teach intent, not syntax (`# We poll every 5 s because the SDK
  doesn't surface a push subscription yet`, not `# increment i`).

## 10. Lab-by-lab implementation guide

### Lab 0 — Setup

**Bicep (`bicep/main.bicep`)** — `targetScope = 'resourceGroup'`. Parameters: `location`
(default `eastus2`), `foundryAccountName`, `foundryProjectName`, `appInsightsName`,
`logAnalyticsName`, `deployments` (array). Resources: AIServices account (api version
`2024-10-01`), project child (`2025-04-01-preview`), `@batchSize(1)` model deployments,
Log Analytics workspace (PerGB2018, 30-day retention), workspace-based Application
Insights. Outputs: `foundryAccountName`, `foundryProjectName`, `foundryEndpoint`,
`projectEndpoint` (= `'${account.properties.endpoint}api/projects/${project.name}'`,
account endpoint already ends with `/`), `appInsightsConnectionString`.

**`main.parameters.json`** — sensible defaults (e.g. `mm-cohere-workshop`,
`mm-cohere-project`, `mm-cohere-appi`, `mm-cohere-la`) plus the four deployments array
matching §3.

**`provision.sh`** — bash, `set -euo pipefail`. Reads/prompts `RESOURCE_GROUP`,
`LOCATION` default `eastus2`. `az group create`, `az deployment group create`. Prints
outputs. Idempotent.

**`setup-env.sh`** — bash, `set -euo pipefail`. Args: `[--force] [RESOURCE_GROUP]`.
Refuses to overwrite `.env` unless `--force`. Looks up subscription id, tenant id,
AIServices account in RG, child project, deployments **by `properties.model.name`**
(so pre-provisioned RGs with different deployment casing still work), App Insights
component name + connection string, and **Foundry account key 1** for the Cohere
ClientV2 vendor env vars. Computes `COHERE_BASE_URL = ${AZURE_AI_ENDPOINT%/}/providers/cohere`.
Copies `sample.env` → `.env` then sed-replaces every variable. Prints a summary table
(mask the API key as `[set]`).

**`verify.sh`** — sources `../.env`. Lists deployments, then makes one inference call
per deployment using `az account get-access-token --resource https://cognitiveservices.azure.com`
and `curl`. Prints PASS/FAIL per deployment.

**`SETUP.md`** — uses the **car-factory analogy** (Foundry = factory, model = engine,
agent = chassis, instructions = manual). Leads with the GitHub Codespaces flow
(devcontainer auto-installs deps; `az login --use-device-code` because Codespaces
cannot open a local browser). Local-machine path is presented as the alternative.
Two Mermaid diagrams: factory topology + provisioning happy path with pre-provisioned
RG branch.

### Lab 1 — Travel concierge with Command A

**Tools (`tools/booking_tools.py`)** — pure-Python functions: `search_flights`,
`search_hotels`, `search_cars` (each with a `book_*` partner). Read from
`data/catalogs/*.json`, filter reasonably, return list[dict]. Docstrings are detailed
because they are read by the agent as tool descriptions.

**Data**
- `data/travel-policy.md` — ~1 page synthetic policy, **numbered rules** so the
  custom evaluator can reason about specific rule violations.
- `data/catalogs/{flights,hotels,cars}.json` — 10–20 records each with fields the
  tools filter on. **Date convention:** all `depart_date`, `return_date`,
  `available_from`, `available_to` values fall in **July–December 2026** to keep
  the synthetic data future-dated; the `04-eval-dataset.jsonl` queries reference
  the same window. Re-spread these dates when re-running the workshop in a future
  year so prompts stay future-dated relative to the run date.

**Custom evaluator (`evaluators/policy_adherence.py`)** — class implementing the
`azure-ai-evaluation` custom-evaluator protocol (`__call__(*, query, response, **kwargs)
-> dict`). Uses **Command A as the judge model** via the OpenAI-compatible client
exposed by `azure-ai-projects` against the Foundry deployment. Returns three keys:
`violates_policy: bool`, `policy_adherence_score: int` (0–5), `reasoning: str`.
Gracefully degrades when `FOUNDRY_PROJECT_ENDPOINT` is unset.

**Tracing helper (`_tracing.py`)** — `configure_tracing()` calls
`azure.monitor.opentelemetry.configure_azure_monitor` using
`APPLICATIONINSIGHTS_CONNECTION_STRING` and instruments httpx. Safe no-op when env
var is unset.

**Notebooks**
1. `01-verify-model.ipynb` — `AIProjectClient.deployments.list()` to confirm
   `command-a` exists; one chat completion via the project's OpenAI-compatible client;
   brief Cohere SDK fallback example.
2. `02-build-agent.ipynb` — create a Foundry **prompt agent** named `travel-concierge`
   backed by `command-a`. Register the booking tools (function-calling). Attach
   `data/travel-policy.md` as grounding (vector store if SDK supports it, else
   inlined into the system prompt — document both). Persist `LAB1_AGENT_ID` back
   into `.env`.
3. `03-test-prompts.ipynb` — 8–10 smoke prompts (one is intentionally a luxury-car
   request to provoke a policy nudge).
4. `04-eval-dataset.jsonl` — 20 rows, fields `{query, expected_intent, axis}`,
   covering quality / safety / agentic / policy.
5. `05-eval-round1-baseline.ipynb` — built-in evaluators (Groundedness, Relevance,
   Coherence, Fluency, ContentSafety, IntentResolution, ToolCallAccuracy,
   TaskAdherence) against baseline generic instructions. Upload to Foundry; print
   summary table + portal-URL pattern.
6. `06-eval-round2-grounded.ipynb` — tighten instructions + attach catalogs +
   policy as grounding; rerun; show delta vs round 1.
7. `07-eval-round3-policy-evaluator.ipynb` — add `PolicyAdherenceEvaluator`; rerun;
   show what it catches that the built-ins missed.
8. `08-redteam-scan.ipynb` — cloud red-team scan via the AI Red Teaming agent in the
   latest SDK. Graceful degradation when `azure-ai-evaluation[redteam]` is not
   installed.
9. `09-load-test.py` — asyncio + httpx. CLI `--rps --duration --failure-ratio`,
   env-overridable. Reads `FOUNDRY_PROJECT_ENDPOINT` + `LAB1_AGENT_ID`. Posts mixed
   prompts (eval-dataset + intentionally malformed ones for failures). Exits with
   `total / success / 4xx / 5xx / p50 / p95`.
10. `10-monitoring-walkthrough.md` — guides the learner through the Foundry
    Monitoring tab after running the load test.

**Bridge (`bridge/travel-capability-hooks.md`)** — five travel "if only my concierge
could…" prompts each mapped to a Lab 2 notebook (2a, 2b, 2c, 2d, 2e).

**README** — concierge system overview + hill-climb-loop Mermaid diagrams.

### Lab 2 — Cohere capabilities

## 11. Implementation order (todos)

Run in this sequence (parallelism in parentheses):

1. **Foundation (sync, do yourself)** — folder skeleton; `cohere/README.md`;
   `cohere/sample.env`; rewrite `cohere/requirements.txt` into commented groups;
   install deps and confirm they resolve.
2. **Three parallel agents** — Lab 0, Lab 1, Lab 2 each as its own background
   agent. They are independent because the foundation is in place.
3. **Sequential follow-ups (each gated on its predecessors)**:
   - Enrich 2b with restored text + image+text fused multimodal search examples.
   - Restore 2a–2e fidelity to vendor (strip the early "Try it Yourself: Travel"
     sections; restore vendor body content; copy vendor `dataset/`,
     `dataset.test/`, `search_images/`; create `chroma_db/` placeholder).
   - Migrate Try-It-Yourself sandbox content into `99-try-it-yourself-travel.ipynb`
     as `From 2x` sub-sections and add cross-link pointers in each 2x notebook.
     Update paths from `assets/travel-*` to `try.travel/...`.
   - Add Mermaid visuals to the four key doc files.
   - Comments pass: add teaching comments to Lab 0/1 code cells, scripts, Python
     modules, and the capstone notebook. Skip 2a–2e (vendor).
   - Plain-language pass: rewrite markdown across Lab 0/1 + the capstone to
     high-school-friendly explanations. Skip 2a–2e markdown.
   - Separator pass: add `---` after `What you will do` in every notebook.
   - Combined Codespaces-first + Cohere-endpoint-reconciliation pass.
